# CS 230 Portfolio – The Gaming Room

## Software Design Document
This repository contains my completed software design document for The Gaming Room project.

## Reflection

### Client Summary
The client for this project was *The Gaming Room*, a company looking to expand their Android-based game, *Draw It or Lose It*, into a web-based application that supports multiple platforms.The goal was to design a scalable system that could handle multiple users, games, teams, and players while ensuring that data such as names and identifiers remained unique. The application also needed to support a client-server architecture so that users could access the game from different devices through a web browser.

### What I Did Well
One area I did particularly well in developing this documentation was clearly organizing the system design and explaining how each component works together. I was able to break down the system into manageable parts, such as the GameService, Game, Team, and Player classes, and explain how they relate to each other. I also did well in evaluating different operating platforms and making a strong recommendation based on performance, cost, and scalability.

### Helpful Design Process
Working through the design document was especially helpful when developing the code because it gave me a clear plan before writing anything. By defining requirements, constraints, and system structure ahead of time, I was able to better understand how the application should function. This made implementing concepts like the singleton pattern and managing relationships between objects much more straightforward.

### What I Would Improve
If I could revise one part of my work, I would improve the Evaluation section by adding more detailed comparisons between platforms, especially in areas like cost, scalability, and long-term maintenance. While I covered the main points, I could strengthen the analysis by including more specific examples and deeper technical reasoning.

### Understanding User Needs
I interpreted the user’s needs by focusing on what the client wanted the system to accomplish, such as supporting multiple users, maintaining unique names, and working across different platforms. I translated those needs into technical requirements and design decisions, such as using a centralized service class and a web-based architecture. Considering the user’s needs is important because it ensures the final product is functional, efficient, and aligned with the client’s goals rather than just being technically correct.

### My Design Approach
When approaching the software design, I focused on breaking the system into smaller components and identifying how they interact. I used object-oriented principles like inheritance, encapsulation, and abstraction to keep the design organized and reusable. In the future, I would continue using these strategies along with planning tools like UML diagrams and design documents to better visualize the system before coding. I would also place more emphasis on scalability and user experience early in the design process to ensure the application can grow effectively.
